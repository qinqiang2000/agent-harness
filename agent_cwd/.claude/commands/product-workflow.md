# /product-workflow — 产品工作流编排器

## 职责

本 command 是 5 步产品工作流的唯一入口，负责：

1. 每步完成后校验产物完整性 + 读取 exit-status，决定下一步
2. 遇到 BLOCKED / NEEDS_HUMAN 时暂停并报告，等待用户指令
3. 全链路完成后输出执行摘要

## 工作流节点总览

五步工作流的通道路由与每步产物一览：

```
① 需求分析 ──→ ② 本体查询与映射 ──→ ③ 本体更新 Gate ──→ ④ 页面设计 ──→ ⑤ PRD 生成
（必经）         （条件执行）            （条件执行）          （条件执行）     （必经）
```

| 步骤        | Skill                | 产物文件名模式                              | 数量        | 下游消费方 |
| --------- | -------------------- | ------------------------------------ | --------- | ----- |
| ①         | requirement-analysis | `{id}_需求分析报告.md`                     | 1         | ②⑤    |
| ② 完整模式    | ontology-context     | `{id}_特性清单.md`                       | 1         | ③④⑤   |
| ② 完整模式    | ontology-context     | `{特性ID}_本体映射报告.md`                   | N（= 子需求数） | ④⑤    |
| ② lite 模式 | ontology-context     | `{id}_本体映射摘要.md`                     | 1         | ③⑤    |
| ③         | ontology-update      | `{特性ID}_本体检查报告.md`                   | N（= 子需求数） | ④⑤    |
| ④         | prototype-design     | `{特性ID}_页面设计说明.md`                   | N（= 子需求数） | ⑤     |
| ⑤ PRD-A   | generate-prd         | `{特性ID}_{title简称}_用户故事设计规格说明书_v*.md` | N（= 子需求数） | 最终交付  |
| ⑤ PRD-B   | generate-prd         | `{id}_{title简称}_用户故事设计规格说明书_v*.md`   | 1         | 最终交付  |

**说明：**
- `{id}` = Linear Issue ID 或用户提供的需求标识符（如 `QUASAF-16`）
- `{特性ID}` = 对象前缀 + 动作序号（如 `F-I-D04`），由 ② 特性清单定义
- ② lite 模式仅在 Fast Lane · 规则类 Bug 时触发
- ② 完整模式走 Standard / Deep Lane；③④ 是否执行取决于通道和覆盖度（详见通道路由表）
- 所有产物统一写入同一个 `output-path` 目录，不分子目录

## 输入参数

| 参数 | 必填 | 说明 |
|------|------|------|
| `$0` | 是 | 需求描述（文本）、本地文档路径、或 Linear issue ID |
| `$1` | 是 | 输出路径（所有 skill 产物统一写入此目录） |
| `--lane=XXX` | 否 | 显式指定通道，透传给 ① |
| `--skip-confirm` | 否 | 跳过中间确认步骤（仅建议熟练用户使用） |

## 通道路由表

① 完成后，编排器读取产物头部的 `lane` + `priority` + 需求分类，按以下路由表决定调用链：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        ① 需求分析（必经）                                │
│                              │                                         │
│                    读取 lane / 需求分类                                  │
│                              │                                         │
│              ┌───────────────┼───────────────┐                         │
│              ▼               ▼               ▼                         │
│         Fast Lane      Standard Lane     Deep Lane                     │
│              │               │               │                         │
│              │          ┌────┴────┐     ┌────┴────┐                    │
│              │          ▼         ▼     ▼         ▼                    │
│              │     新增/扩展/   Bug/体验/  新增/扩展/   Bug/体验/          │
│              │     配置变更    性能优化   配置变更    性能优化              │
│              │          │         │     │         │                     │
│              │          ▼         │     ▼         │                     │
│              │         ②→③→⑤     │    ②→③→④→⑤   │                     │
│              │                    ▼               ▼                     │
│              │                   ⑤              ④→⑤                    │
│              ▼                                                         │
│     Bug·规则类 → ②(lite)→③(条件)→⑤                                    │
│     Bug·体验类/技术类 → ⑤(lite)                                        │
│     非需求 → 关闭                                                       │
│     小改动 → 按需 ⑤                                                     │
└─────────────────────────────────────────────────────────────────────────┘
```

### 路由决策矩阵

| lane     | 需求分类                | 调用链                    | 说明                        |
| -------- | ------------------- | ----------------------- | ------------------------- |
| fast     | Bug · 规则类           | ①→②(lite)→③(条件)→⑤      | 本体匹配判断规则命中/缺失，③ 按结论决定是否执行 |
| fast     | Bug · 体验类 / 技术类     | ①→⑤(lite)               | 无需本体映射，直接简化 PRD           |
| fast     | 非需求                 | ①                       | 关闭，不继续                    |
| fast     | 小改动                 | ①→⑤(lite)               | 按需简化 PRD                  |
| standard | 新增/扩展/配置变更          | ①→②→③→⑤                 | 标准链路，④ 跳过                 |
| standard | Bug · 规则类           | ①→②(lite)→③(条件)→⑤      | 同 fast 规则类 Bug 路径         |
| standard | Bug · 体验类/技术类/性能优化  | ①→⑤(lite)               | 无需本体映射                    |
| standard | 体验优化                | ①→④→⑤(lite)             | 需要页面设计                    |
| deep     | 新增/扩展/配置变更          | ①→②→③→④→⑤               | 完整链路                      |
| deep     | Bug · 规则类           | ①→②(lite)→③(条件)→⑤      | 同 fast 规则类 Bug 路径         |
| deep     | Bug · 体验类/技术类/性能优化  | ①→⑤(lite)               | 无需本体映射                    |
| deep     | 体验优化                | ①→④→⑤(lite)             | 需要页面设计                    |

## 执行协议

### Step 0 · 参数校验

1. 校验 `$0` 和 `$1` 是否提供
2. 确认 `$1` 输出目录存在（不存在则创建）
3. 检查输出目录下是否已有产物（有则提示：继续 / 覆盖 / 终止）

### Step 1 · 调用 ① 需求分析

```
invoke: 01-requirement-analysis
args: $0 $1 [--lane=$lane]
```

**完成后校验：**

1. 产物文件存在：
   - Fast: `{output-path}/*_需求分析报告.md`（精简版）
   - Standard/Deep: `{output-path}/*_需求分析报告.md`
2. 产物头部包含 `lane` 和 `priority` 字段
3. 读取 `lane` / `priority` / 需求分类 / 推荐路径

**路由决策：**

- 读取需求分类，查路由决策矩阵，确定后续调用链
- 向用户报告：
  ```
  ① 完成。
  · lane: {lane} | priority: {priority}
  · 需求分类: {category}
  · 后续调用链: {chain}
  是否继续？[继续 / 调整通道 / 终止]
  ```
- `--skip-confirm` 时跳过确认，直接继续

### Step 2 · 调用 ② 本体查询与映射（条件执行）

**准入条件：** 调用链包含 ②

```
invoke: 02-ontology-context
args: {①产物路径} $1 [profile]
```

**完成后校验：**

1. 判断模式：
   - 若 `bug-subtype=rule`（lite 模式）→ 校验 `{output-path}/*_本体映射摘要.md` 存在，读取 `bug-ontology-result`
   - 否则（完整模式）→ 校验 `{output-path}/*_特性清单.md` 存在，特性清单中每个子需求都有对应的 `*_本体映射报告.md`
2. 读取覆盖度汇总（✅/⬜/📝 统计，完整模式）或 bug-ontology-result（lite 模式）
3. **完整模式额外校验**：读取 `mvp-scope.threshold-triggered`，若为 `true` → 确认用户已完成人工介入确认，且 P0 范围已锁定

**路由决策：**

- 完整模式：
  - 读取 `mvp-scope.threshold-triggered`：
    - `true` → 后续 ③→⑤ 仅处理 P0 特性（`mvp-scope.p0-count` 个），P1/P2/P3 特性跳过
    - `false` → 全量特性进入后续流程
  - 读取覆盖度，查路由决策矩阵，确定后续调用链
- lite 模式：读取 `bug-ontology-result`，按以下决策：
  - `HIT` → 跳过 ③，直接进 ⑤
  - `PARTIAL` → 提示用户可选执行 ③
  - `MISS` → 必须执行 ③
- 向用户报告：
  ```
  ② 完成。
  · lane: {lane} | bug-subtype: {bug-subtype}
  · 模式: {完整模式 / lite 模式}
  · 结论: {覆盖度摘要 / bug-ontology-result}
  · 本期 MVP 范围: P0 特性 {n} 个 / 延后特性 {n} 个（threshold-triggered: {true/false}）
  · 后续调用链: {chain}（仅 P0 特性）
  是否继续？[继续 / 调整通道 / 终止]
  ```
- `--skip-confirm` 时跳过确认，直接继续

### Step 3 · 调用 ③ 本体更新 Gate（条件执行）

**准入条件：** 调用链包含 ③ 且 ② exit-status = `DONE_WITH_GAPS`

③ 执行后，编排器读取产物头部的 `coverage-rate`、`coverage-tier`、`gaps-remaining`，按三档分支处理：

```
读取 ③ 产物
    │
    ├─ coverage-tier = HIGH（>= 80%）且 gaps-remaining = 0
    │      → exit-status: DONE，静默放行，继续 ④/⑤
    │
    ├─ coverage-tier = HIGH（>= 80%）且 gaps-remaining > 0
    │      → 向用户报告（HIGH 档文案），提供 [a][b][c]
    │
    ├─ coverage-tier = MID（60-79%）
    │      → 向用户报告（MID 档文案），提供 [a][b][c]
    │
    └─ coverage-tier = LOW（< 60%）
           → 向用户报告（LOW 档文案），仅提供 [a][c]，强制阻断
```

**HIGH 档（coverage-rate >= 80%，gaps-remaining > 0）向用户报告：**

```
③ Gate 检查完成。
· 本体覆盖度：{n}%（HIGH ✅）· 仍缺失项：{gaps-remaining} 项

以下本体缺失项尚未补齐：
  ⬜ {对象}.{缺失类型}: {缺失路径} — {期望内容}

覆盖度已达 80% 以上，可以继续，缺失项将在 PRD 6.4 节标注。

请选择：
  [a] 人工补齐后重新检查（我会等待你的指令后重新执行 ③）
  [b] 继续生成 PRD（PRD 6.4 节将标注 {gaps-remaining} 项待补充）
  [c] 终止工作流
```

**MID 档（60% <= coverage-rate < 80%）向用户报告：**

```
③ Gate 检查完成。
· 本体覆盖度：{n}%（MID ⚠️）· 仍缺失项：{gaps-remaining} 项

以下本体缺失项尚未补齐：
  ⬜ {对象}.{缺失类型}: {缺失路径} — {期望内容}

覆盖度在 60%-80% 之间，允许继续，但 PRD 和原型中将标注待补充内容，
建议事后补充本体后重新生成。

请选择：
  [a] 人工补齐后重新检查（我会等待你的指令后重新执行 ③）
  [b] 继续生成 PRD（PRD/原型将标注 {gaps-remaining} 项待补充，建议事后补本体）
  [c] 终止工作流
```

**LOW 档（coverage-rate < 60%）向用户报告（强制阻断，无 [b] 选项）：**

```
③ Gate 检查完成。
· 本体覆盖度：{n}%（LOW ❌）· 仍缺失项：{gaps-remaining} 项

覆盖度低于 60%，无法继续生成 PRD 和原型。
本体基础不足时生成的 PRD 可靠性极低，强制阻断以避免产出无效文档。

必须先补充以下本体缺失项，使覆盖度达到 60% 以上：
  ⬜ {对象}.{缺失类型}: {缺失路径} — {期望内容}

请选择：
  [a] 人工补齐后重新检查（我会等待你的指令后重新执行 ③）
  [c] 终止工作流
```

**各选项后续行为：**

| 选项 | HIGH 档 | MID 档 | LOW 档 |
|------|--------|--------|--------|
| [a] | 等待，用户说"继续"后重新执行 ③ | 同左 | 同左 |
| [b] | exit-status → `DONE`，编排器继续 ④/⑤ | exit-status → `DONE_PARTIAL`，编排器继续 ④/⑤ | 不提供 |
| [c] | 输出执行摘要并终止 | 同左 | 同左 |

### Step 4 · 调用 ④ 页面设计（条件执行）

**准入条件：** 调用链包含 ④ 且当前特性 `interaction_type = Page`

- 如特性清单中无任何 `Page` 类型特性 → 整体跳过 Step 4
- 逐个模式下：仅对 `interaction_type = Page` 的特性调用 ④，其余跳过
- 无特性清单时（简单需求）：读取 ① 的实现形态倾向判断，仅 `Page` 时执行

```
invoke: 04-prototype-design
args: {需求标识} $1 [--profile=XXX]
```

**完成后校验：**

1. `{output-path}/*_页面设计说明.md` 存在
2. 页面数量符合 Less-is-More 上限
3. 五区完整度符合 priority 矩阵

### Step 5 · 调用 ⑤ PRD 生成

**始终执行**（所有调用链的终点）

**生成模式判断：**

```
读取特性清单 mvp-scope.p0-count（如有特性清单）

IF 无特性清单（纯 Bug/体验/性能）:
  单次调用 ⑤，生成 1 份 PRD-B
  invoke: 05-generate-prd
  args: $1 [--context=$1]

IF p0-count ≤ 5:
  批量模式：一次性传入所有 P0 特性上下文
  invoke: 05-generate-prd
  args: $1 [--context=$1]

IF p0-count > 5:
  逐个模式：按依赖拓扑排序，每次传入 1 个特性的上下文
  对每个 P0 特性:
    invoke: 05-generate-prd
    args: $1 [--context=$1] [--feature={特性ID}]
  无依赖的特性可并行调用
  每个 PRD 生成后即可独立存档触发 N1 Stage B
```

> **注意**：逐个模式下，③④⑤ 均按单个特性独立调用。
> 如调用链包含 ④，④ 也按逐个特性调用（与 ⑤ 同步）。

**完成后校验：**

1. PRD 文件存在（PRD-A 或 PRD-B）
2. 模板占位符全部替换
3. 第四章标题与 interaction_type 一致：
   - Page → 「页面交互规格」且内容非空（④ 产物已内联）
   - API → 「接口规格」
   - Batch → 「批处理规格」
   - Event → 「事件规格」
4. 如有特性清单 → PRD 数量与子需求数量一致（逐个模式下逐步累计）

## 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 任一 skill 产物校验失败 | 暂停，报告缺失项，询问用户：重试 / 跳过 / 终止 |
| ③ NEEDS_HUMAN | 暂停等待用户指令（见 Step 3） |
| 用户选择终止 | 输出已完成步骤的摘要，保留已生成产物 |
| 通道升级触发 | ① 的自动升级钩子触发后，重新计算调用链并通知用户 |

## 重要约束

1. **编排器不做业务逻辑**：每步的具体执行由对应 skill 负责，编排器只做路由、校验、报告
2. **lane/priority 单一来源**：以 ① 产物头部为准，编排器不自行判断
3. **不跳过用户确认**：除非 `--skip-confirm`，每个关键节点都需用户确认
4. **产物路径统一**：所有 skill 产物写入同一个 `$1` 目录，不分子目录
5. **幂等安全**：重复执行时检测已有产物，提示用户选择
6. **不自行降级通道**：编排器不得把 Deep 降为 Standard
7. **摘要必输出**：无论正常完成还是中途终止，都输出执行摘要
