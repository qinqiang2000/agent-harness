# PRD 生成

## 定位

本 skill 是产品工作流的第 5 步（最后一步），负责将前 4 步的输出组装为最终的产品规格说明书。

```
产品工作流
═══════════════════════════════════════════════════════════
  ① 需求分析 → ② 本体查询与映射 → ③ 本体更新 Gate → ④ 页面设计 → ⑤ PRD 生成
                                                            （本 skill）
  各步输出 ──────────────────────────────────────────────────→ 汇总组装
```

**职责边界：**
- **做什么**：读取上游产物 → 选择路径（A 完整 / B 简化）→ 为当前目标特性组装一份 PRD → 质量自检 → 输出
- **不做什么**：不调用本体 MCP、不做 JTBD 分析、不修改本体、不设计原型、不批量生成多份 PRD

**原子性说明：**
- 每次调用只处理一个目标特性，输出一份 PRD
- 批量循环由编排层（product-workflow）负责

## 输入参数

- `$0` — 输出路径（生成 PRD 的目标目录，必填）
- `--context=<dir>`（可选）— 上游产物所在目录。如不指定，skill 会在当前对话上下文中查找

## 上游产物说明

本 skill 消费以下上游 skill 的输出：

| 来源 | 产物 | 文件名模式 | 必须/可选 | 用途 |
|------|------|-----------|---------|------|
| ① 需求分析 | JTBD 分析报告 | `*_需求分析报告.md` | **必须** | 用户故事、业务场景、需求分类、推荐路径、业务需求列表、优先级判断 |
| ② 本体查询与映射 | 特性清单 | `*_特性清单.md` | 可选（上下文参考） | 了解当前特性在整体需求中的位置和依赖关系 |
| ② 本体查询与映射 | 当前特性映射报告 | `{特性ID}_本体映射报告.md` | 条件必须 | PRD 的核心数据源——本体语义、产品映射、覆盖度评估（PRD-A 时必须） |
| ③ 本体更新 Gate | 本体检查报告 | `{特性ID}_本体检查报告.md` | 可选 | 第六章 6.4 节的缺失项和建议调整项 |
| ④ 页面设计 | 页面设计说明 | `{特性ID}_页面设计.md` | 条件必须（interaction_type=Page 时） | 内联到第四章「页面交互规格」 |

> **最低启动条件**：
> - 必须有 ① 的 JTBD 分析报告（含需求分类）
> - 如果需求分类为「新增业务能力 / 扩展现有能力 / 配置变更」→ 还必须有 ② 的 特性清单 + 映射报告
> - 如果需求分类为「Bug 修复 · 规则类」→ 有 ②(lite) 产物（`{id}_本体映射摘要.md`），生成简化 PRD 时在变更说明中引用 bug-ontology-result 和命中规则 ID
> - 如果需求分类为「Bug 修复 · 体验类 / 技术类 / 体验优化 / 性能优化」→ 不需要 ②，直接基于 ① 的分析报告生成简化 PRD
>
> **当前阶段说明**：基于本体驱动的 PRD-A 路径当前均为 API 或 Event 类型，不涉及 Page 类特性，因此 ④ 产物当前不会被消费。第四章直接走 API/Event 模板，无需 ④ 输入。待后续本体覆盖 Page 类需求时再启用。

## 执行流程

严格按以下阶段顺序执行。

---

### 阶段 1 · 收集上游产物

**1.0 上游产物校验**

按 [validate-upstream 协议](../commands/validate-upstream.md) 执行：
1. 读取本 skill frontmatter 中的 consumes 声明
2. 在 `{output-path}/` 或 `--context` 目录下逐项校验存在性 + 必填字段 + exit-status
3. PASS → 继续；FAIL → 终止并报告

**1.1 扫描产物**

1. 如果指定了 `--context=<dir>`，从该目录扫描上游产物文件
2. 如果未指定，在当前对话上下文中查找
3. 识别并读取各上游产物：
   - 通过文件名模式匹配（见上表）
   - 记录找到和缺失的产物
4. 从 ① 需求分析报告中提取：
   - 需求分类与推荐路径
   - 用户故事（As a / I want / So that）
   - 业务需求列表（如有）
   - 优先级判断
5. **确定当前目标特性**：
   - 编排层传入的当前特性 ID（从特性清单中获取）
   - 无特性清单时（简单需求）→ 将 ① 的整体需求作为单个目标
6. **校验必须产物**：
   - 必须有 ① 的 JTBD 分析报告
   - PRD-A 时必须有当前特性的映射报告
   - interaction_type=Page 时必须有 ④ 页面设计产物（当前本体流程不涉及，暂不触发）

---

### 阶段 2 · 组装 PRD

根据当前目标特性的性质判断走完整路径（A）还是简化路径（B）：

**路径判断规则：**
- 有 `{特性ID}_本体映射报告.md` → PRD-A（完整 PRD），使用模板 [templates/TEMPLATE.md](templates/TEMPLATE.md)
- 有 `{id}_本体映射摘要.md`（规则类 Bug lite 产物）→ PRD-B（简化 PRD），在变更说明中引用 `bug-ontology-result` 和命中规则 ID，使用模板 [templates/TEMPLATE_LITE.md](templates/TEMPLATE_LITE.md)
- 无 ② 产物（体验类/技术类 Bug / 体验优化 / 性能优化等）→ PRD-B（简化 PRD），使用模板 [templates/TEMPLATE_LITE.md](templates/TEMPLATE_LITE.md)

> 无特性清单时（简单需求），直接走 PRD-B。

输出质量对标 [templates/EXAMPLE_F-I-D04.md](templates/EXAMPLE_F-I-D04.md)，按需查阅，不强制全文加载。

---

#### 共性章节（PRD-A / B 通用）

以下两章结构一致，数据来源按优先级回退：有 ② 映射报告优先用 ②，无则用 ①。

**一、特性概述**

| 字段 | PRD-A 数据来源 | PRD-B 数据来源 |
|------|---------------|---------------|
| What | ② 映射报告「业务场景描述」提炼 | ① Functional Jobs 提炼 |
| 特性类型 | 从动作定义推导 | 需求分类（Bug 修复 / 体验优化 / 性能优化） |
| 覆盖对象/动作 | 直接引用 ② 映射报告（PRD-B 不填此项） | — |
| 用户价值 | ② 映射报告推导，1-2 句 | ① Gains 分析推导 |
| 适用范围 | ② Profile Applicability 推导 | ① 业务场景推导 |
| 成功指标 | Primary / Secondary / Guardrail | 同左 |
| 不做什么 | 结合 ① 优先级判断，明确排除项 | 同左 |

**二、用户故事**

- **业务场景**：
  - 优先从 ① JTBD 报告的 Functional Jobs 和业务场景提炼
  - 如无 ① → 从 ② 映射报告的「业务场景描述」生成
- **主故事**：As a / I want / So that 三段式
  - 优先引用 ① 的用户故事
  - 如无 ① → 从 ② 映射报告推导
- **用户旅程**：
  - **PRD-A**：
    - Happy Path：从 ② 映射报告的「Happy Path 要素」组装（Guard → 函数链 → Effect）
    - Unhappy Path：从 ② 映射报告的「Unhappy Path 要素」组装（至少三类）
  - **PRD-B**：
    - Happy Path：基于 ① 的分析描述期望的正常流程
    - Unhappy Path：基于 ① 的 Pains / Common Mistakes 推导异常场景
  - 如有 ④ 页面设计说明 → 在旅程步骤中引用页面设计
- **验收标准**：
  - 功能 AC（F-AC-XX）：PRD-A 基于状态迁移，PRD-B 基于期望行为
  - 业务规则 AC（R-AC-XX / C-AC-XX）：仅PRD-A，基于 ② 映射报告的规则包
  - 性能 AC（P-AC-XX）：如有性能要求（PRD-B 性能优化类必填）

---

#### PRD-A 扩展章节（完整 PRD 专属）

以下四章仅PRD-A 生成，PRD-B 跳过。

**三、校验规则说明**

数据来源：② 映射报告 → 关联规则包

- **3.1 规则包调用**：直接引用 ② 映射报告的规则列表（VL1/VL2/VL3/VL4）
  - 如本动作无校验规则 → 填"本动作不执行校验规则"
  - 如有 MVP 外规则 → 单独列出
- **3.2 规则包外的约束**：从 ② 映射报告的 Guard 和异常处理提取
- **3.3 执行策略与状态结果**：从 ② 映射报告的状态迁移组装

> 规则包内规则以规则 ID 索引为准，不在 PRD 中重复描述逻辑。

**四、[按 interaction_type 动态切换]**

第四章标题和内容根据当前特性的 `interaction_type` 动态选择：

| interaction_type | 第四章标题 | 数据来源 |
|-----------------|-----------|---------|
| API | 接口规格 | ② 映射报告 → 关联属性 + 状态定义 |
| Page | 页面交互规格 | ④ 页面设计产物（内联） |
| Batch | 批处理规格 | ② 映射报告 → 触发条件 + 处理逻辑 |
| Event | 事件规格 | ② 映射报告 → 触发事件 + 处理逻辑 |

**interaction_type = API 时：**

- **Guard 前置条件**：从 ② 映射报告的 Guard 提取
- **4.1 请求参数**：
  - 从 ② 映射报告的「关联属性」转换为接口字段
  - 字段命名对齐本体属性路径驼峰
  - 平铺单表，区分调用入参和平台内部读取字段
- **4.2 响应规格**：按场景分表（成功/失败/409/500）
- **4.3 响应示例**：每种场景一个 JSON 示例

**interaction_type = Page 时：**

- **4.1 页面结构**：页面布局、区域划分、组件选型
- **4.2 交互流程**：用户操作步骤、状态流转、反馈机制
- **4.3 字段与组件**：表单字段、校验规则、联动逻辑
- **4.4 异常与边界**：空状态、加载态、错误提示

> 内容来源：④ 页面设计产物（`{特性ID}_页面设计.md`），直接内联到本章。

**interaction_type = Batch 时：**

- **4.1 触发条件**：定时规则 / 手动触发 / 事件触发
- **4.2 处理逻辑**：批量处理流程、分批策略、并发控制
- **4.3 输入输出**：数据源、处理结果、日志记录
- **4.4 异常处理**：失败重试、部分成功、告警通知

**interaction_type = Event 时：**

- **4.1 触发事件**：上游事件来源、触发条件
- **4.2 处理逻辑**：事件消费流程、幂等保证
- **4.3 数据变更**：状态迁移、字段回写
- **4.4 异常处理**：消费失败、重试策略、死信处理

**五、地区差异说明**

数据来源：② 映射报告 → Profile Applicability

- 按 Profile 列出差异点
- 如无差异 → 填"本特性逻辑全球通用，无地区差异"

**六、附录：产品本体对应实现关系**

数据来源：② 映射报告 → 覆盖度评估 + 文档引用索引；③ 本体检查报告（如有）

- **6.1 本体映射总览**：直接引用 ② 映射报告的映射总览，标注 ✅/⬜
- **6.2 执行函数链**：直接引用 ② 映射报告的函数链 ASCII 图
- **6.3 本体文档引用索引**：直接引用 ② 映射报告的引用索引
- **6.4 本体待补充与建议调整清单**：
  - 从 ② 映射报告的覆盖度评估提取 ⬜ 缺失项和 📝 建议调整项
  - 如有 ③ 本体检查报告 → 合并其中的建议
  - 如 ③ coverage-tier=MID，在本节开头追加醒目标注：
    ```
    > ⚠️ 当前本体覆盖度为 {n}%（MID），以下缺失项已知晓并允许继续，
    > 第 3 章规则包和第 4 章接口字段中的对应内容为推断值，
    > 待本体补充后必须复核。
    ```
  - **醒目标注：PRD 不具备本体修改权限，所有建议需人工审核后方可执行**

---

#### PRD-B 扩展章节（简化 PRD 专属）

以下两章仅PRD-B 生成，替代PRD-A 的三~六章。

**三、变更说明**

数据来源：① JTBD 分析报告

- **当前行为**：描述现有系统的行为（Bug 修复：错误行为；体验优化：当前体验；性能优化：当前性能指标）
- **期望行为**：描述修复/优化后的目标行为
- **影响范围**：列出受影响的模块、页面或接口
- **回归风险**：标注可能受影响的相关功能，供测试参考

**四、附录**

- **关键假设与风险**：引用 ① 的「关键假设与风险」章节
- **信息不足说明**：引用 ① 的「信息不足说明」章节（如有）

---

### 阶段 3 · 质量自检

对生成的 PRD 进行一次完整扫描，对照以下 checklist 逐条标注通过/不通过，不通过项立即修正。

**通用自检项（两条路径均适用）：**

1. **模板完整性**：PRD-A 六章 / PRD-B 四章，结构完整无遗漏
2. **用户故事一致性**：主故事是否与 ① 的分析结论一致
3. **验收标准可测性**：每条 AC 独立可测，有明确输入与预期输出
4. **上游产物溯源**：每章标注数据来源（①/②/③/④），确保可追溯
5. **模板占位符全部替换**：不得残留 `[特性ID]`、`SXX`、`[对象名]` 等占位符
6. **删除填写提示**：模板中的 `> 💡 **填写提示**` 块在最终输出中删除

**PRD-A 额外自检项：**

7. **状态一致性**：所有状态引用包含完整格式 `SXX（中文名 / 英文枚举值）`，不得仅写 `SXX`
8. **规则包对齐**：3.1 节规则 ID 列表与 ② 映射报告一致
9. **字段命名规范**：接口字段对齐本体属性路径驼峰命名
10. **函数链完整性**：6.2 节函数链覆盖 L1→L2→L3→L4 各层（如适用）
11. **本体引用索引**：6.3 节列出所有引用的本体文档路径
12. **示例质量**：参考 EXAMPLE_F-I-D04.md 的详细程度，确保输出质量不低于该水平
13. **第四章标题一致性**：第四章标题与 interaction_type 一致（API→接口规格 / Page→页面交互规格 / Batch→批处理规格 / Event→事件规格）

**PRD-B 额外自检项：**

14. **变更说明清晰度**：当前行为 vs 期望行为描述清晰、可验证
15. **影响范围完整性**：列出所有受影响的模块

### 阶段 4 · 输出

**⚠️ 文件命名强制约束（不可违反）：**
- PRD-A：`{output-path}/{特性ID}_{动词+宾语}（{场景}）_产品规格说明书_v{major}.{minor}.md`
- PRD-B：`{output-path}/{需求标识}_{动词+宾语}（{场景}）_产品规格说明书_v{major}.{minor}.md`
- 禁止使用 `prd.md`、`spec.md`、`product-spec.md` 等其他名称

**文件命名规则：**
- `动词+宾语`：从特性清单 title 列（PRD-A）或 ① 需求分析报告标题（PRD-B）提取核心动词+宾语
- `场景`：括号内标注业务场景（如"成品油"、"不动产租赁"）
- 版本号：首次生成 v1.0，迭代修改时递增 minor（v1.1, v1.2...），重大重写递增 major
- PRD-A 示例：`F-I-D04_校验发票合规性（成品油）_产品规格说明书_v1.0.md`
- PRD-B 示例：`QUASAF-16_修复按钮样式（开票页）_产品规格说明书_v1.0.md`
- ❌ 反例：`F-I-D04_成品油校验_用户故事设计规格说明书_v1.0.md`（宾语缺失、场景混入动作、后缀过长）

**⚠️ 产物数量强制约束（不可违反）：**
单次执行单份产出。本 skill 每次调用只写入 1 个 PRD 文件，禁止创建任何其他文件。

1. 确认输出目录存在（不存在则创建）
2. 生成当前目标特性的 PRD：
   - PRD-A：`{output-path}/{特性ID}_{动词+宾语}（{场景}）_产品规格说明书_v1.0.md`
   - PRD-B：`{output-path}/{需求标识}_{动词+宾语}（{场景}）_产品规格说明书_v1.0.md`
3. 向用户报告：
   - 生成的 PRD 文件路径和路径类型（A/B）
   - 上游产物使用情况（哪些已消费、哪些缺失）
   - PRD-A 覆盖度摘要（✅/⬜/📝 各多少项，如适用）
   - 如有待补充章节 → 提示用户补充对应上游产物后重新生成

### skill-exit-status 规范

PRD 产物头部必须包含：

```yaml
skill-exit-status: DONE | DONE_WITH_GAPS | BLOCKED
skill-exit-detail: {补充说明}
```

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `DONE` | PRD 生成完毕，质量自检通过 | 无残留占位符 |

## 重要约束

**通用约束（两条路径均适用）：**

1. **不调用本体 MCP**：本 skill 不直接访问本体
2. **不做需求分析**：用户故事内容来自 ① 或 ② 的推导，不重新做 JTBD
3. **模板占位符全部替换**：不得残留 `[特性ID]`、`SXX`、`[对象名]` 等占位符
4. **删除填写提示**：模板中的 `> 💡 **填写提示**` 块在最终输出中删除

**PRD-A 专属约束：**

5. **不修改本体**：PRD 是本体的消费者。6.4 节只能给出建议，由人工审核后决定
6. **规则包以本体为准**：3.1 节规则信息仅作快速索引，如有不一致以 RulePackage 为准
7. **不重复描述规则逻辑**：规则包内规则以规则 ID 索引为准
8. **字段命名对齐本体**：接口字段名对齐本体属性路径驼峰
9. **状态必须完整**：所有状态引用必须包含 `SXX（中文名 / 英文枚举值）`
10. **6.4 区分两类问题**：「本体缺失项」（⬜）和「建议调整项」（📝）分开列出
11. **单次执行单份产出**：每次调用只生成一份 PRD，批量循环由编排层负责

**PRD-B 专属约束：**

12. **变更说明必须具体**：当前行为和期望行为必须描述到可验证的程度，不得笼统
13. **不引入本体概念**：简化 PRD 面向非本体场景，不使用本体术语（对象/动作/状态码等）
14. **禁止产出额外文件**：本 skill 每次调用只写入 1 个 PRD 文件，禁止创建任何其他文件
