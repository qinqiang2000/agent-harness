# 产品上下文查询与映射

## 定位

本 skill 是产品工作流的第 2 步，负责将 ① 的业务需求映射到本体对象+动作，拆解为最小交付单元（特性清单），对每个单元提取完整语义上下文，并评估本体覆盖度。

```
产品工作流
═══════════════════════════════════════════════════════════
  ① 需求分析 → ② 本体查询与映射 → ③ 本体更新 Gate → ④ 页面设计 → ⑤ PRD 生成
                   （本 skill）
                      │
                      ▼
               特性清单（最小单元清单）
             + 对象决策记录（反例 + 举一反三）
             + 各动作的本体映射报告
             + 覆盖度评估（✅/⬜/📝）
                  （供 ③④⑤ 消费）

  通道准入：
  ┌──────────┬──────────────┬──────────────────────────────────────┐
  │ 通道     │ 是否进入 ②   │ 执行模式                              │
  ├──────────┼──────────────┼──────────────────────────────────────┤
  │ Fast     │ ⚠️ 条件进入  │ 仅 bug-subtype=rule 时进入，走 lite 模式 │
  │ Standard │ ✅ 进入      │ 完整模式                              │
  │ Deep     │ ✅ 进入      │ 完整模式                              │
  └──────────┴──────────────┴──────────────────────────────────────┘
```

**职责边界：**
- **做什么**：读取 JTBD 报告 → 粗扫本体识别对象+动作 → 对象决策（复用/扩展/新增） → 拆解 特性清单 → 逐个精读映射 → 覆盖度评估
- **不做什么**：不做需求分析（① 的职责）、不修改本体（③ 的职责）、不生成 PRD（⑤ 的职责）

## 输入参数

- `$0` — ① 的 JTBD 分析报告路径，或业务需求描述文本
- `$1` — 输出路径（必填）
- `$2`（可选）— Profile 名称，默认 `CN-LEQI`

## 执行流程

### 阶段 0：通道与级别准入校验

**0.1 上游产物校验**

按 [validate-upstream 协议](../commands/validate-upstream.md) 执行：
1. 读取本 skill frontmatter 中的 consumes 声明
2. 在 `{output-path}/` 下逐项校验存在性 + 必填字段 + exit-status
3. PASS → 继续；FAIL → 终止并报告

**0.2 通道校验**

1. 从 `{output-path}/` 下查找 ① 的产物，读取头部 `lane`、`priority`、`bug-subtype` 字段
2. **分流处理**：
   - `lane: fast` 且 `bug-subtype: rule` → **进入 lite 模式**（跳转至阶段 L）
   - `lane: fast` 且 `bug-subtype` 为 `ux` / `tech` 或未填 → **立即终止**，提示：
     ```
     ⚠️ 本 issue 在 ① 中被分流为 Fast Lane（非规则类 Bug），不进入 ② 本体映射。
     · 体验类/技术类 Bug → 请直接执行 ⑤ 生成简化 PRD
     · 非需求 → 已在 ① 关闭
     · 小改动 → 按需直接走 ⑤
     ```
   - `lane: standard` 或 `lane: deep` → 进入完整模式（阶段 1）
   - 未找到 ① 产物 → 提示先执行 ①，终止
3. 记录 `priority`（P0/P1/P2/P3），用于后续阶段的详略裁剪
4. 将 lane / priority / bug-subtype 原样透传到本 skill 产物头部

### 阶段 1：读取输入与业务需求识别

1. 如果 `$0` 是文件路径 → 读取 JTBD 分析报告
2. 如果 `$0` 是文本 → 作为业务需求描述
3. 缺参 → 提示调用格式并终止
4. 从输入中提取业务需求列表：
   - 优先从 JTBD 报告的「业务需求列表」章节提取（① 的标准输出）
   - 每条业务需求为「动词+宾语+情境」格式
   - 如果报告中没有明确的业务需求列表，从 Functional Jobs 中推导
5. 向用户确认提取到的业务需求列表

---

### 阶段 2：粗扫本体 + 对象决策

**2.1 识别相关对象**

根据业务需求关键词，判断可能涉及的本体对象。读取对象注册表 [references/object-registry.md](Product%20Skill%20-ontology/02-ontology-context/references/object-registry.md)，结合业务需求中的关键词匹配：

| 关键词示例 | 可能涉及的对象 |
|-----------|-------------|
| 发票/开具/开票/校验/上报 | invoice-data |
| 红字确认单/红冲申请 | credit-note-apply |
| 电子发票文件/PDF/版式 | invoice-file |
| 开票申请/申请单 | billing-request |
| 销方/购方/参与方 | party |

关键词匹配完成后，**必须进一步通过 LinkTypes 发现关联对象**：

1. 对每个已命中的对象，通过本体访问服务获取 `{abbr}_02_LinkTypes.md`
2. 读取 LinkTypes 中的关联对象，将其纳入对象集
3. 对新纳入的关联对象递归执行步骤 1-2（深度 ≤ 2 层，避免无限扩展）
4. 记录每个对象的发现来源（关键词命中 / LinkType 关联）

> 示例：invoice-data 通过 LinkType 关联 invoice-file（开具成功后生成电子文件）和 credit-note-apply（红票依赖红字确认单），若只靠关键词匹配会遗漏这两个对象。

**2.2 获取动作速查表**

对每个识别出的对象（含 LinkType 发现的关联对象），通过本体访问服务获取 `{abbr}_04_ActionTypes.md`（动作速查表），不获取完整 6 类文件。

**2.3 匹配业务需求→对象+动作**

将每条业务需求与本体动作速查表匹配：
- 一条业务需求可能对应多个动作
- 一条业务需求可能跨多个对象

**2.4 对象复用 / 扩展 / 新增决策**

对每条业务需求，按决策树走完一遍：

```
            ┌─────────────────────────────────────────────────┐
            │ 问：该业务需求本质上是对哪个业务实体的操作？       │
            └────────────────┬────────────────────────────────┘
                             ▼
           ┌─────────────────────────────────────┐
           │ Q1: 现有对象的现有动作集合能否承载？   │
           └─────────────────┬───────────────────┘
                 ┌───────────┴─────────────┐
                YES                        NO
                 │                          │
                 ▼                          ▼
          【复用】X + ACT-XX    ┌─────────────────────────────┐
                                │ Q2: 能否通过新增动作承载？     │
                                └────────────┬───────────────┘
                                  ┌──────────┴────────────┐
                                 YES                      NO
                                  │                        │
                                  ▼                        ▼
                       【扩展】X + 新增 ACT    ┌─────────────────────────┐
                                              │ Q3: 能否通过对象组合承载？ │
                                              └────────────┬────────────┘
                                                ┌──────────┴────────────┐
                                               YES                      NO
                                                │                        │
                                                ▼                        ▼
                                     【组合】X + Y + LinkType       【新增】新对象 Z
```

**每个决策必须记录**：
- 决策路径（Q1 YES / Q2 YES / Q3 YES / 全 NO）
- 对象类型标注：复用 / 扩展 / 新增
- **反例记录**：选定一个分支后，明确列出其他分支为什么被否决

**反例记录条数（按 priority 裁剪）**：

| 级别 | 反例记录 |
|------|---------|
| P0 | 完整记录全部 2-3 条被否决分支 + 详细否决理由 |
| P1 | 完整记录全部被否决分支 |
| P2 | ≥2 条被否决分支 |
| P3 | ≥1 条被否决分支 |

### 阶段 3：输出 特性清单 + 对象集收敛

**3.1 特性清单（最小单元清单）**

| id | title | object | action | priority | is_mvp | blocked_by | group | interaction_type |
|----|-------|--------|--------|----------|--------|------------|-------|-----------------|
| {ID}-01 | [动作名称（场景）] | [对象名] | ACT-XX | P0/P1/P2/P3 | true/false | [依赖的子需求 ID] 或 — | [大需求方向] | Page/API/Batch/Event/Other |

**列说明：**
- `id`：Feature ID，格式 `F-{对象缩写}-{动作编号}[-后缀]`，贯穿全流程唯一标识
- `title`：动词+宾语+场景说明
- `object`：本体对象名（如 invoice-data、credit-note-apply）
- `action`：本体动作 ID（如 ACT-01、ACT-04）
- `priority`：P0/P1/P2/P3
- `is_mvp`：由优先级推导（P0=true，P1/P2/P3=false）
- `blocked_by`：依赖的 feature ID，多个用逗号分隔，无依赖填 `—`
- `group`：大需求方向标签（如"正数发票开具"、"增值税校验"），用于 Linear Issue 标签分组
- `interaction_type`：实现形态（Page/API/Batch/Event/Other），决定 ④ 是否执行及 ⑤ PRD 第四章结构

**interaction_type 精判规则：**

从本体动作定义中推断实现形态：

| 推断依据 | interaction_type | 典型动作 |
|---------|-----------------|---------|
| 动作由用户在界面上主动触发，有表单输入 | Page | 创建、编辑、提交申请 |
| 动作由系统/外部系统调用，无用户界面 | API | 校验、查询、数据同步 |
| 动作由定时任务或批量触发 | Batch | 批量导入、定时对账 |
| 动作由上游对象状态变更自动触发 | Event | 回写结果、状态同步、通知 |
| 无法明确判断 | Other | — |

> 优先级：本体动作定义 > ① 粗判结果 > 默认 Other
> 如同一动作可能有多种触发方式，以主要使用场景为准。
> 仅 `Page` 类型会触发 ④ 页面设计，其余类型跳过 ④。
> **当前阶段说明**：基于本体驱动的需求（通过 ② 映射的特性）当前均为 API 或 Event 类型，暂无 Page 类特性。④ 页面设计路径保留但当前不会触发。

**优先级语义（特性级，以此为最终判断，① 为初步判断）：**

| 级别 | 含义 | 对下游影响 |
|------|------|----------|
| P0 | 不做需求走不通的最小闭环，缺一不可 | 本期必须进入 ③→⑤ |
| P1 | 重要补充，核心路径可跑通但需求不完整 | 超阈值时本期延后，下期优先 |
| P2 | 可独立延后，不影响核心验收 | 下一期再走工作流 |
| P3 | 本期明确不做或探索性 | 保留在特性清单，标注"本期不做" |

> ① 的优先级是基于业务价值和紧急度的初步判断；② 的优先级是基于实现路径依赖分析的最终判断，原则一致，以 ② 为准。

**拆解原则：**
- 每个子需求 = 一个对象的一个动作（最小可独立交付单元）
- 按对象生命周期排序（创建→校验→上报→回写）
- 标注依赖关系
- 跨对象的关联通过依赖关系体现

**Fast Lane 专用规则（优先于上述原则）：**
- Fast Lane 原则上不拆分子需求，1 个 issue = 1 个子需求
- ① 输出的多条业务需求描述视为同一实现的不同角度（如校验逻辑、错误处理、适用范围），合并为 1 个子需求
- 所有来源业务需求全部保留在「来源业务需求」字段中，不得丢失
- 仅当多条业务需求明确对应不同对象或不同动作（可独立上线、有不同依赖）时，才允许拆分，并在子需求备注中注明拆分理由

**3.2 最小对象集收敛与自检**

1. **去重**：同一对象被多条需求引用，合并为一次记录
2. **最小性自检**：
   - 对象数量 ≤ 业务需求数量 / 2（超过需有充分理由）
   - 每个新增对象必须至少承载 2 个动作
   - 每个新增对象必须至少有 1 条 LinkType 连回已有对象
   - 无游离对象

**3.3 举一反三清单**

对每个选中的对象，列出同一对象集还能服务哪些未来场景：

| 级别 | 举一反三条数 |
|------|------------|
| P0 | ≥3 条未来场景 |
| P1 | ≥2 条未来场景 |
| P2 | ≥1 条未来场景 |
| P3 | 允许跳过（仅可行性结论） |

**3.4 输出并等待用户确认**

展示：特性清单 + 依赖关系图（ASCII）+ 对象决策记录 + 反例 + 举一反三 + 最小性自检结果

**依赖关系图要求：**
- 每个节点标注特性ID + 优先级（如 `F-I-D04（P0）`）
- 体现 P0 关键路径（主干链路）
- 跨对象依赖用虚线或注释区分

**阈值检查（展示后、用户确认前执行）：**

```
特性总数 ≤ 5 → 正常确认流程，用户确认后进入阶段 4
特性总数 > 5 → 强制暂停，输出人工介入报告
```

**人工介入报告格式（特性总数 > 5 时输出）：**

```
⚠️ 特性清单超过阈值（{n} > 5），需人工确认后才能继续。
（注：Deep Lane 复杂需求超过阈值属正常情况）

P0 特性（本期必做，共 {n} 个）：
  · {特性ID} — {标题}
  · ...

P1 特性（建议延后，共 {n} 个）：
  · {特性ID} — {标题}
  · ...

P2/P3 特性（本期不做，共 {n} 个）：
  · {特性ID} — {标题}
  · ...

请确认：
  [a] P0 范围是否正确？（确认后仅 P0 特性进入 ③）
  [b] 是否需要将不同对象的特性拆分为独立的 Linear 父 issue？
  [c] P1/P2/P3 特性保留在特性清单标注"延后"，下期再走工作流
  [d] 确认 P0 范围，继续执行阶段 4-6（精读 + 映射报告生成）
```

> **[d] 选项说明：** 选择 [d] 仅确认 P0 范围并继续后续阶段（精读 + 映射报告输出），**不立即改变 stage**。stage 变更为 `confirmed` 的时机：
> 1. ② 全部完成（阶段 6 输出后），特性清单 + 所有映射报告均已就绪
> 2. 产品经理最终审阅特性清单 + 所有映射报告
> 3. 确认无误后手动将 `stage` 改为 `confirmed` 并存档到 Git
> 4. 此时 N1 工作流引擎检测到 `confirmed` 状态，触发父子 Issue 骨架创建（映射报告已就绪，子 Issue 正文不会为空）
>
> 如果工作流引擎未部署，`confirmed` 状态仅作为产品确认标记，不影响 skill 后续执行。

用户确认 P0 范围后进入阶段 4，**仅 P0 特性进入后续精读和映射**；P1/P2/P3 特性保留在特性清单，标注"延后"或"本期不做"，不进入本期 ③→⑤ 流程。

### 阶段 4：精读本体（获取完整文件 + 提取语义）

对 特性清单 中涉及的每个对象，通过本体访问服务获取完整 6 类本体文件：

| # | 文件 | 层级 | 定义 |
|---|------|------|------|
| 1 | `{abbr}_01_Properties.md` | 语义层 | 属性 |
| 2 | `{abbr}_02_LinkTypes.md` | 语义层 | 链接类型 |
| 3 | `{abbr}_03_ValueSets.md` | 语义层 | 状态枚举 |
| 4 | `{abbr}_04_ActionTypes.md` | 动力层 | 动作类型（阶段 2 已获取，复用） |
| 5 | `{abbr}_05_Functions/{abbr}_05.0_Functions.md` | 动力层 | 业务逻辑函数 |
| 6 | `{abbr}_05_Functions/{abbr}_05.1_{Profile}_RulePackage.md` | 动力层 | 规则包 |

**处理规则：**
1. 文件不存在则记录缺失
2. 同一对象只获取一次

**对 特性清单 中每个动作，提取 9 类本体语义：**

1. **动作基本信息**：动作 ID、名称、描述
2. **Guard（前置状态约束）**：允许的起始状态列表
3. **Effect（状态迁移）**：成功/失败/异常目标状态
4. **Profile Applicability**：该动作在目标 Profile 下的可用性
5. **关联函数**：函数链（L1→L2→L3→L4）
6. **关联规则包**：按 VL1/VL2/VL3/VL4 分层的规则列表
7. **关联属性**：该动作读/写的属性字段
8. **关联链接**：涉及的对象间链接
9. **状态定义**：涉及的状态码定义

### 阶段 5：应用方法论映射 + 覆盖度评估

读取 [references/methodology.md](Product%20Skill%20-ontology/02-ontology-context/references/methodology.md)，对每个动作执行：

**5.1 产品概念映射**

| PM 概念 | 本体概念 | 映射方式 |
|---------|---------|---------|
| 业务场景 | 动作描述 + 上下文 | 解释触发背景 |
| 用户操作 | Action（动作） | 1个特性 = 1个动作 |
| 业务规则 | Guard + Function + RulePackage | 前置条件 + 函数链 + 校验规则 |
| 系统反馈 | Effect + State | 状态迁移 + 属性写回 |
| 数据字段 | Property | 关键传参与返回字段 |
| 异常场景 | 失败状态节点 | 状态机中的失败分支 |
| 对象联动 | 状态事件触发 + LinkType | 上游状态变更触发下游 |

**5.2 本体覆盖度评估**

逐一检查每个属性、函数、规则、状态在本体文件中的定义情况：
- 本体中已定义 → ✅
- 本体中未定义（缺失）→ ⬜，记录到覆盖度清单
- 已有但可能需要调整 → 📝，记录并附理由

### 阶段 6：输出

**输出产物：**

**⚠️ 文件命名强制约束（不可违反）：**
- 特性清单：`{output-path}/{需求标识}_特性清单.md` — 禁止使用 `feature-list.md`、`features.md` 等其他名称

## Lite 模式执行流程（Fast Lane · 规则类 Bug 专用）

> 由阶段 0 识别 `bug-subtype: rule` 后跳转至此，跳过完整模式的阶段 1-6。

### 阶段 L1 · 提取 Bug 关键词

1. 从 ① 需求分析报告中提取 Bug 描述
2. 识别关键词：校验字段名、金额类型、状态名、规则描述等
3. 根据关键词判断可能涉及的本体对象（参考对象注册表）

### 阶段 L2 · 轻量本体搜索

对识别出的对象，通过本体访问服务搜索以下三类文件（不获取完整 6 类）：

| 搜索目标 | 文件 | 搜索内容 |
|---------|------|---------|
| 规则包 | `{abbr}_05.1_{Profile}_RulePackage.md` | 关键词匹配规则 ID、规则名称、规则描述 |
| 状态定义 | `{abbr}_03_ValueSets.md` | 关键词匹配状态码、状态名 |
| 动作前置条件 | `{abbr}_04_ActionTypes.md` | 关键词匹配 Guard 条件 |

### 阶段 L3 · 输出匹配结论

**三种结论：**

| 结论 | 含义 | 触发条件 | PRD 定位 |
|------|------|---------|---------|
| `HIT` | 规则命中 | 本体中找到匹配规则/状态/Guard | "实现 {规则ID}，对齐本体定义" |
| `PARTIAL` | 部分命中 | 本体有相关定义但不完整 | "实现 {规则ID}，并补充 {缺失部分}" |
| `MISS` | 未命中 | 本体中无相关定义 | "先补本体，再实现" |

**⚠️ 文件命名强制约束（不可违反）：**
输出文件名必须为 `{output-path}/{id}_本体映射摘要.md`，禁止使用其他名称。

**⚠️ 产物数量强制约束（不可违反）：**
lite 模式只允许写入 **1 个文件**：`{id}_本体映射摘要.md`，禁止创建任何其他文件。

写入 `{output-path}/{id}_本体映射摘要.md`：

### 阶段 L4 · 路由决策（决定是否执行 ③）

| bug-ontology-result | ③ 是否执行 | 说明 |

## 重要约束

1. **始终获取最新本体**：每次执行都通过本体访问服务获取最新版本
2. **只读不写**：本 skill 不修改本体内容，只提取和映射
3. **不管理认证**：token、API key 等凭据由本体访问服务管理，skill 不接触
4. **状态必须完整**：所有状态引用必须包含 `SXX（中文名 / 英文枚举值）`
5. **LinkType 文档第三部分**：仅 invoice-data 对象保留重点链接局部说明 + UBL 映射，其他对象按需输出
6. **覆盖度评估必须执行**：完整模式不可跳过，这是 ③ 的关键输入
7. **粗扫与精读分离**：阶段 2 只获取 ActionTypes，阶段 4 才获取完整 6 类文件
8. **特性清单 需用户确认**：阶段 3 输出后须等待用户确认
9. **报告是中间产物**：供 ③④⑤ 消费的结构化上下文
10. **Fast Lane 规则类 Bug 走 lite 模式**：阶段 0 识别出 `bug-subtype: rule` 跳转 lite 模式；其他 Fast Lane 立即终止
11. **lane / priority / bug-subtype 透传不得丢失**：产物头部必须含这三个字段
12. **反例强制**：完整模式对象决策必须附反例记录，否则重做
13. **最小对象集优先**：能复用不扩展，能扩展不新增，能组合不新建
14. **举一反三按级别裁剪**：条数严格按 3.4 矩阵执行
15. **单一职责**：每个对象只代表一个业务实体，禁止拼装对象
16. **禁止产出额外文件**：本 skill 只允许写入 `produces` 声明中的文件，禁止创建任何其他文件。执行完成后 output-path 下新增文件数量：完整模式 = 1 + N（N = 子需求数量），lite 模式 = 1（仅 `{id}_本体映射摘要.md`）。
