# 第 ④ 步 · 页面设计（Less is More）

## 0. 在五步工作流中的定位

```
产品工作流
═══════════════════════════════════════════════════════════
  ① 需求分析 → ② 本体查询与映射 → ③ 本体更新 Gate → ④ 页面设计 → ⑤ PRD 生成
                                                 （本 skill）
                                                     │
                                                     ▼
                                          {特性ID}_页面设计.md
                                         （中间产物，被 ⑤ 内联到第四章）
```

**准入条件：**
- 当前特性的 `interaction_type = Page`（从特性清单中获取）
- 如 interaction_type 非 Page（API / Batch / Event / Other），本 skill 不应被调用（由编排层保证）
- 无特性清单时（简单需求）：读取 ① 需求分析报告的「实现形态倾向」，仅 Page 时执行

> **当前阶段说明**：基于本体驱动的需求当前均为 API 或 Event 类型，暂无 Page 类特性。本 skill 路径保留但当前不会被触发。

**职责边界：**
- **做什么**：基于映射报告的属性和状态信息，设计页面结构、交互流程、字段组件
- **不做什么**：不做需求分析、不查询本体、不生成 PRD、不输出最终交付物、不产出 UI 设计稿图片

**原子性说明：**
- 每次调用处理一个特性，输出一份页面设计文件
- 产物定位为中间产物，被 ⑤ 内联到 PRD 第四章「页面交互规格」，不独立对外交付
- 批量循环由编排层负责

## 1. 输入参数

| 参数 | 必填 | 说明 |
|------|------|------|
| `$0` | 是 | 当前特性 ID（如 F-I-D01） |
| `$1` | 是 | 输出路径 |
| `--profile=XXX` | 否 | Profile 名称（默认 `CN-LEQI`） |

**可选上下文：**
- 需求分析报告（① 产物）— 理解用户场景和交互需求
- 特性清单（② 产物）— 理解当前特性在整体需求中的位置
- 当前特性的映射报告（② 产物）— Properties / ValueSets / ActionTypes

## 2. 执行流程

### 阶段 0 · 准入校验

1. 确认当前特性 `interaction_type = Page`
   - 从特性清单中读取当前特性的 interaction_type
   - 非 Page → 终止，提示"非 Page 类型特性不产出页面设计"
2. 读取 `lane` 和 `priority`：
   - `lane: fast` → 终止，提示"Fast Lane 不产出页面设计"
   - 识别到 `severity` → 终止（Bug 不进入 ④）

### 阶段 1 · 消费上游产物

1. 必读文件：
   - ② 当前特性的 `{特性ID}_本体映射报告.md`（Properties / ValueSets / ActionTypes / 函数链）
   - ① `*_需求分析报告.md`（用户角色、业务需求列表）
2. 可选文件：
   - ② `*_特性清单.md`（理解当前特性在整体需求中的位置）
   - ③ 本体检查报告（若有 ⬜ 缺失项已补齐）
3. 映射报告缺失 → 终止并提示

### 阶段 2 · 三类角色识别与诉求矩阵

基于 ① 的用户角色分析，锁定三类基准角色：

| 角色代码 | 角色名 | 典型诉求 | 关注字段/状态 |
|---------|--------|---------|-------------|
| ROLE-MGT | 管理层 | 全局数据看板、异常早发现、合规风险掌控 | 总量、异常数、合规风险标识、趋势 |
| ROLE-LEAD | 部门负责人 | 团队粒度数据、审批/分派、进度掌控 | 按团队筛选、待审核数、流程卡点 |
| ROLE-BIZ | 业务操作人员 | 单据录入、操作执行、状态查询 | 单据明细、操作按钮、状态流转 |

对每条 MVP 业务需求，标注涉及的角色集合，产出诉求矩阵。

### 阶段 3 · Less-is-More 页面收敛

**收敛规则（优先级从高到低）：**

1. **R1 · 同对象同级合并**：同一对象的不同动作合并到一个页面的"操作区"
2. **R2 · 多角色同页分视图**：同一对象的不同角色诉求用视角切换（Tab/角色模式）
3. **R3 · 状态看板合并明细**：对象状态看板作为明细列表的上方卡片区
4. **R4 · 跨对象链接走抽屉**：通过 LinkType 关联的对象走侧抽屉/弹窗
5. **R5 · 写操作优先同页**：动作表单在同页侧抽屉或 Modal

**页面数量上限**：
- 对象数 ≤ 2：≤ 1 个核心页面
- 对象数 3-4：≤ 2 个核心页面
- 对象数 ≥ 5：≤ 3 个核心页面（须给出不能再合并的反例）

**"一页满足性测试"**：
- 管理层打开能看到：完整数据状态看板 + 关键异常
- 部门负责人打开能看到：按部门过滤的全部视图 + 待办
- 业务员打开能看到：我负责的单据 + 所有可执行动作

### 阶段 4 · 页面骨架设计（按 priority 裁剪五区完整度）

| 级别 | 状态看板区 | 筛选区 | 列表区 | 侧抽屉 | 动作表单区 |
|------|----------|--------|--------|--------|----------|
| P0 | 完整+趋势+异常高亮 | 完整+高级筛选 | 完整+行级动作 | 完整+LinkType导航+时间线 | 完整+Guard提示 |
| P1 | 完整+异常高亮 | 完整 | 完整 | 完整 | 完整 |
| P2 | 完整 | 基础（时间+部门+状态） | 完整 | 基础信息+操作区 | 完整 |
| P3 | 聚合卡片 | 基础（时间） | 基础列表 | 基础信息 | 复用行操作 |

每个页面包含 5 个区：

```
┌─────────────────────────────────────────────────────────────┐
│ [页面名称]                                      [角色切换]    │
├──────────────────────────────────────────────────────────────┤
│ 【状态看板区】对象 ValueSets 全状态聚合                        │
│  ┌──────┬──────┬──────┬──────┬──────┐                        │
│  │ S01  │ S02  │ S03  │ S12  │ S13  │← 状态码 + 中文名       │
│  │ 草稿 │校验失败│校验成功│开具成功│开具失败│                    │
│  │ 120  │ 35   │ 218  │ 180  │ 6    │← 当期计数 + 趋势       │
│  └──────┴──────┴──────┴──────┴──────┘                        │
├──────────────────────────────────────────────────────────────┤
│ 【筛选区】 [时间][部门][Profile][状态多选][高级筛选]            │
├──────────────────────────────────────────────────────────────┤
│ 【列表区】                                      [+ 新建]      │
│ │编号│客户│金额│状态│最后操作│操作列                            │
│ │....│....│....│....│........│[校验][上报][更多]               │
├──────────────────────────────────────────────────────────────┤
│ 【侧抽屉】选中行后展示：                                      │
│  · 基础信息（Properties 全量）                                │
│  · 关联对象（LinkType 导航）                                  │
│  · 状态时间线（ValueSets 流转历史）                            │
│  · 操作按钮区（Guard 允许的 ActionTypes）                      │
├──────────────────────────────────────────────────────────────┤
│ 【动作表单区】侧抽屉/Modal 内的操作表单                        │
└──────────────────────────────────────────────────────────────┘
```

### 阶段 4.5 · AI 兜底降级页面（仅 Deep + tech-path=ai-agent/hybrid）

> Standard 跳过；Deep 且 tech-path=traditional 也跳过。

三类降级场景：

| 降级场景 | 触发条件 | 页面呈现 | 用户动作 |
|---------|---------|---------|---------|
| AI 处理中 | AI 调用 >3s | 进度条 + "跳过AI手动填写"按钮 | 等待 / 放弃AI |
| AI 置信度低 | 置信度 < 阈值 | "AI推荐"+"人工填写"双栏 | 复核/修改/提交 |
| AI 失败兜底 | 超时/下线/幻觉 | 错误提示 + 自动切传统表单 | 传统表单继续 |

hybrid 路径侧抽屉双栏布局：

```
┌─── 侧抽屉 · AI 助模式 ──────────────────────────┐
│ [AI 推荐] | [人工填写] ← Tab 切换                 │
├──────────────────────────────────────────────────┤
│ AI 推荐栏：                                       │
│   · 字段 A：建议值 X   [采纳] [修改] [忽略]       │
│   · 置信度：0.92       [查看依据]                 │
├──────────────────────────────────────────────────┤
│ 人工填写栏（传统表单）：                           │
│   · 字段 A：[         ]                          │
├──────────────────────────────────────────────────┤
│ [全部采纳 AI]  [仅保留人工]  [取消]  [提交]       │
└──────────────────────────────────────────────────┘
```

### 阶段 5 · 字段可视化矩阵（Properties × 角色 × 状态）

基于 ② 映射报告的 Properties 与 ValueSets，对每个对象给出字段矩阵：

| 字段 | 类型 | ROLE-MGT | ROLE-LEAD | ROLE-BIZ | S01必填 | S02只读 | 脱敏 | 来源 |
|------|------|---------|----------|---------|--------|--------|------|------|
| invoiceNumber | string | ✅ | ✅ | ✅ | 系统生成 | 是 | - | 本体 |
| buyerTaxId | string | 部分脱敏 | 部分脱敏 | ✅ | 是 | 是 | 中间6位 | 本体 |

### 阶段 6 · 动作可见性矩阵（ActionTypes × 角色 × 状态）

基于 ② 映射报告的 ActionTypes + Guard，给出动作可见性矩阵：

| 动作 | 动作名 | Guard | ROLE-MGT | ROLE-LEAD | ROLE-BIZ |
|------|--------|-------|---------|----------|---------|
| ACT-04 | 提交合规校验 | S01, S02 | 查看 | 审批 | 执行 |
| ACT-12 | 上报开具 | S03 | 查看 | 审批 | 执行 |

**权限判定公式**：
- 按钮可见性 = 角色权限 ∩ 当前状态 ∈ Guard
- 按钮可点击性 = 可见 ∧ 无并发锁 ∧ 无未完成前置审批
- 按钮置灰时必须给出 tooltip

### 阶段 7 · 生成页面设计说明

**⚠️ 文件命名强制约束（不可违反）：**
输出文件名必须为 `{output-path}/{特性ID}_页面设计.md`，禁止使用 `ui-design.md`、`prototype.md`、`design.md` 等其他名称。

**⚠️ 产物数量强制约束（不可违反）：**
单次执行单份产出。本 skill 每次调用只写入 1 个页面设计文件，禁止创建任何其他文件。

对当前特性，写入 `{output-path}/{特性ID}_页面设计.md`，结构：

```markdown
# {特性ID} 页面设计说明

> **lane**：standard / deep
> **priority**：P0-P3
> **tech-path-hint**：traditional / ai-agent / hybrid
> **五区完整度**：完整 / 精简 / 骨架-only
> **三角色差异**：完整矩阵 / 简表 / 跳过
> **设计原则**：Less is More
> **上游输入**：② 特性清单 + 映射报告 / ③ 本体更新结果
> **页面数量**：N

## 一、诉求矩阵与角色模型
## 二、页面收敛决策
## 三、页面清单（逐页五区骨架 + 三角色差异）
## 四、字段可视化矩阵
## 五、动作可见性矩阵
## 六、状态看板设计详述
## 七、可访问性与响应式
## 八、给 ⑤ 的交付说明
```

**MID 档覆盖度标注（coverage-tier=MID 时执行）：**

在产物 YAML header 追加：
```yaml
ontology-coverage-warning: "覆盖度 {n}%（MID），以下内容待本体补充后复核：{缺失项列表}"
```

```
> ⚠️ 本体覆盖度 {n}%（MID）：以下字段/动作在本体中尚未定义，

> {逐项列出 gaps-remaining 中的缺失项}
```

### 阶段 8 · 质量自检

#### 两通道共同必做

1. 页面数量符合启发式上限
2. 五区按 priority 矩阵完整
3. 状态看板覆盖 ValueSets 全部状态
4. 字段矩阵 Properties 全量（P3 允许跳过）
5. 动作矩阵 ActionTypes 全量（P3 允许跳过）
6. 三角色差异表（P0/P1 完整，P2 简表，P3 可跳过）
7. Less-is-More 反例记录
8. 所有字段/状态/动作引用指回 ② 映射报告
9. 无像素级样式规范
10. lane + priority 双透传

#### Deep + AI 额外必做

11. 三类 AI 降级页面齐全
12. hybrid 双栏布局完整（采纳/修改/忽略留痕）
13. AI 中间状态进入状态看板
14. 置信度阈值显性引用
15. AI 失败可一键回落传统表单

### skill-exit-status 规范

页面设计说明产物头部必须包含：

```yaml
skill-exit-status: DONE | DONE_WITH_CONCERNS | BLOCKED
skill-exit-detail: {补充说明}
quality-check:
  total: {n}
  passed: {n}
  failed: {n}
```

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `DONE` | 质量自检全部通过 | 阶段 8 自检项全部通过 |
| `DONE_WITH_CONCERNS` | 完成但有瑕疵 | P3 骨架模式下部分自检项按规则跳过；或 Deep+AI 降级页面设计需用户复核 |
| `BLOCKED` | 无法继续 | ② 映射报告缺失、Fast Lane 禁入、Bug 识别到 severity |

## 4. 重要约束

1. **Fast Lane 禁入**：阶段 0 识别出 Fast 立即终止
2. **lane / priority 透传强制**：产物头部必须继承
3. **Deep + AI 必设降级页面**：跳过等于允许 AI 上线无兜底 UI
4. **不自行改 tech-path**：本 skill 不得擅自把 AI 改成 traditional
5. **Less is More 强制**：页面数上限必须遵守
6. **一页满足三角色**：默认目标，确需分页须证明生命周期不同
7. **状态看板必须完整**：ValueSets 全部状态进入看板
8. **不生成像素规范**：交给 UI 设计师
9. **不出 UI 图片**：只交付 .md
10. **所有引用必须有本体追溯**：指向 ② 映射报告的具体段落
11. **无本体不出设计**：② 映射报告缺失则终止
12. **键盘可达 + 非颜色依赖**：可访问性底线
13. **AI 输出必可编辑**：禁止只读 AI 结果
14. **AI 留痕强制**：采纳/修改/忽略行为必须记录
15. **校验不因 AI 豁免**：提交时 RulePackage 仍完整校验
16. **Bug 不进入 ④**：识别到 severity 立即终止
17. **P3 骨架最低红线**：必须给出核心入口 + 主要跳转路径
18. **级别不可自调**：不得降级或升级处理
19. **禁止产出额外文件**：本 skill 每次调用只写入 1 个文件（`{特性ID}_页面设计.md`），禁止创建任何其他文件
