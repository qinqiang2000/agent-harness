# 需求分析

## 定位

本 skill 是产品工作流的统一入口（第 1 步），负责理解用户真实需求意图，判断需求类型，输出结构化的 JTBD 分析报告，并推荐后续执行路径。

```
产品工作流
═══════════════════════════════════════════════════════════
  ① 需求分析 → ② 本体查询与映射 → ③ 本体更新 Gate → ④ 页面设计 → ⑤ PRD 生成
  （本 skill）
      │
      ├── Fast Lane   → Bug·规则类→②(lite)→③(条件)→⑤ / Bug·体验类/技术类→⑤ / 非需求→关闭 / 小改动→按需⑤
      ├── Standard    → ②→③→(④)→⑤ 标准链路
      └── Deep        → ②→③→④→⑤ 完整链路

  分流产物矩阵：
  ┌──────────┬──────────────────────┬────────────────────────┬──────────────────────┐
  │ 通道     │ Fast（极简）         │ Standard（标准）        │ Deep（深度）         │
  ├──────────┼──────────────────────┼────────────────────────┼──────────────────────┤
  │ 典型场景 │ Bug/文案/权限/小改动 │ 已有对象的扩展/新字段   │ 新对象/新 Profile/AI │
  │ 执行阶段 │ A→A+→A++→B1→C→E     │ A→A+→A++→B(全)→C→D→E  │ A→A+→A++→B(全)→C→D→E│
  │ 产物文件 │ 1 份：需求分析报告.md│ 1 份：需求分析报告.md   │ 1 份：需求分析报告.md│
  │          │ （精简版，lane=fast）│                        │                     │
  │ 下游触发 │ Bug·规则类→②(lite)→③(条件)→⑤ / 体验类/技术类→⑤ │ ②→③→(④)→⑤ │ ②→③→④→⑤ │
  └──────────┴──────────────────────┴────────────────────────┴──────────────────────┘
```

**职责边界：**
- **做什么**：理解用户真实需求意图（Jobs / Pains / Gains）、挑战隐含假设、给出优先级判断、分类需求并推荐后续路径、输出业务需求列表
- **不做什么**：不查询本体（② 的职责）、不修改本体（③ 的职责）、不生成 PRD（⑤ 的职责）、不替代真实用户访谈、不做系统级功能拆解

## 输入参数

| 参数 | 必填 | 说明 |
|------|------|------|
| `$0` | 是 | 需求描述（文本）或本地文档路径，或 Linear issue ID |
| `$1` | 是 | 输出路径（生成报告的目标目录） |
| `--lane=XXX` | 否 | 显式指定通道（fast/standard/deep），覆盖自动分流 |

## 执行流程

### 阶段 1 · 获取需求输入与模式判断

**1.1 获取输入**

1. 如果 `$0` 匹配 Linear issue ID 格式 → 通过 Linear MCP 读取标题、描述、标签、评论
2. 如果 `$0` 是本地文件路径 → 读取文件内容
3. 如果 `$0` 是文本 → 直接作为需求描述
4. 缺参 → 提示调用格式并终止

**1.2 判断输入模式**

| 模式 | 特征 | 处理策略 |
|------|------|---------|
| 明确问题 | 用户/场景/痛点都清晰 | 直接进入分析 |
| 已有方案 | 描述的是解决方案而非问题 | 先回溯到问题层 |
| 模糊直觉 | 感觉有问题但说不清 | 提出 1 个澄清问题 |
| 约束驱动 | 从限制条件出发 | 识别约束背后的真实目标 |

---

### 阶段 2 · 分流决策（Triage）

**目的**：在任何分析前先判定通道。不让小需求跑大流程，也不让大需求被简化。

#### 2.1 分流决策树

```
Q1: --lane 是否显式传入？
    是 → 直接采用，跳 2.2
    否 → Q2

Q2: 是否为明确的 Bug / 文案 / 权限点 / 简单调整？
    判定信号（命中任一即是）：
      · Linear 标签含 bug / typo / docs / copy / permission
      · 标题含"修复 / 报错 / 写错 / 文案 / 权限"等
      · 描述聚焦于单一症状（≤100 字、指向单字段/单按钮）
      · 用户主动声明"小改动 / 简单调整"
    是 → 【Fast Lane】
    否 → Q3

Q3: 是否含"复杂信号"（命中任一即是）？
    · 涉及新增对象（现有对象字典外的业务概念）
    · 涉及新 Profile / 地区 / 税制 / 合规体系
    · 跨 ≥2 个现有对象的联动
    · 跨 ≥2 个角色
    · 战略级/方向性问题
    · 合规/税控/审计高度敏感（失败代价量级 ≥10 万元）
    是 → 【Deep Lane】
    否 → 【Standard Lane】
```

#### 2.2 通道声明与用户覆盖

1. 向用户明确声明分流结果和判定依据
2. 询问用户是否接受（接受 / 升级 / 降级）
3. 用户覆盖时记录 override-reason

#### 2.3 自动升级钩子

即便分流为 Fast 或 Standard，若在后续阶段发现"复杂信号"，必须暂停并提示用户是否升级。

#### 2.4 通道执行规则

| 阶段 | Fast Lane | Standard Lane | Deep Lane |
|------|-----------|---------------|-----------|
| 阶段4 · JTBD | 仅 B1（用户角色 1-2 句） | 完整 B1-B4 | 完整 B1-B4 |
| 价值判定 | 执行（依据仅需 B1） | 执行 | 执行 |
| 业务需求列表 | 跳过 | 执行 | 执行 |
| 输出 | 需求分析报告.md（精简版） | 需求分析报告.md | 需求分析报告.md |

---

### 阶段 3 · 优先级与严重度判定

**目的**：定级别——决定后续阶段的"详略预算"。

#### 3.1 需求优先级（P0 / P1 / P2 / P3）

| 业务价值 ↓ / 紧急度 → | 立即（本 Sprint） | 近期（本季度） | 远期（跨季度） |
|---|---|---|---|
| 核心业务阻断 / 合规刚性 / 战略级 | P0 | P0 | P1 |
| 显著价值提升 / 关键客户要求 | P1 | P1 | P2 |
| 体验优化 / 非关键增量 | P2 | P2 | P3 |
| 探索性 / 可有可无 | P3 | P3 | P3 |

**判定强制引用**：业务价值必须引用 JTBD B4 Gains + B3 Pains；紧急度必须引用 Linear 标签或业务事件。

#### 3.2 Bug 严重度（SEV1-SEV4）

仅当判定为 🐛 Bug 时使用。五维判定：影响范围 / 业务损失 / workaround / 合规风险 / 数据正确性。

#### 3.3 通道 × 级别一致性校验

| 组合 | 合法性 | 处理 |
|------|--------|------|
| Fast + P0 | ❌ 不合法 | 强制升级通道 |
| Fast + SEV1 | ❌ 不合法 | 强制切换 HOTFIX |
| Standard + P0 | ⚠️ 建议升级 | 提示用户 |
| Deep + P3 | ⚠️ 不建议 | 提示 ROI 偏低 |

#### 3.4 级别 × 详略裁剪矩阵（供下游 skill 查阅）

| 阶段 / Skill 环节 | P0 | P1 | P2 | P3 |
|---|---|---|---|---|
| ① JTBD 深度 | B1-B4 完整 | B1-B4 完整 | B1-B4 标准 | B1 + 简化 B3 |
| ② 举一反三 | ≥3 条 | ≥2 条 | ≥1 条 | 允许跳过 |
| ② 反例记录 | 完整 | 完整 | ≥2 条 | ≥1 条 |
| ③ 本体覆盖 | 6 层必补 | 6 层必补 | 可精简 | Properties + ActionTypes |
| ⑤ PRD 章节 | 6 章完整 | 6 章完整 | 6 章 | 允许仅第一+四章 |
| ⑤ 灰度策略 | 强制 | 强制 | 可选 | 可选 |

#### 3.5 向产物头部写入字段

```yaml
lane: fast | standard | deep
priority: P0 | P1 | P2 | P3           # 需求用
severity: SEV1 | SEV2 | SEV3 | SEV4   # Bug 用
priority-rationale: {判定依据}
override-reason: {若有通道/级别覆盖}
```

#### 3.6 实现形态倾向（粗判）

基于需求描述中的用户交互线索，判断整体实现形态倾向：

| 实现形态 | 判断线索 | 说明 |
|---------|---------|------|
| Page | 用户操作界面、表单填写、列表查看、按钮点击、页面跳转 | 有用户直接交互的界面 |
| API | 系统间调用、接口对接、数据传输、无用户界面 | 纯后端接口，无 UI |
| Batch | 定时执行、批量处理、夜间跑批、数据迁移 | 定时/批量任务 |
| Event | 状态变更触发、消息通知、回调、异步处理 | 事件驱动，无用户主动操作 |
| Other | 以上均不明确 | 需 ② 精判后确定 |

> 此为粗判，基于需求描述推断。如需求涉及多种形态，标注主要形态 + 备注。
> 最终以 ② 特性清单中逐特性标注的 `interaction_type` 为准。
> 仅 `Page` 类型会触发 ④ 页面设计，其余类型跳过 ④。

### 阶段 4 · JTBD 需求意图理解

> **按通道执行**：Fast 仅 B1；Standard / Deep 完整 B1-B4。

读取 [references/jtbd-guide.md](Product%20Skill%20-ontology/01-requirement-analysis/references/jtbd-guide.md) 获取分析框架。

**B1 · 识别用户角色**
- 谁是主要用户？（角色、职责、使用场景）
- 在什么情境下触发？
- 是否多角色？诉求差异？

**B2 · 三层 Jobs**（Standard / Deep）
- **Functional Jobs**：动词+宾语+情境，每个附第一性原理回溯链
- **Social Jobs**：在他人面前的形象
- **Emotional Jobs**：希望获得或避免的情绪

**B3 · Pains**（Standard / Deep）
- Challenges / Costliness / Common Mistakes / Unresolved Problems
- 每个标注严重程度（高/中/低）

**B4 · Gains**（Standard / Deep）
- Expectations / Savings（量化）/ Adoption Factors / Life Improvement

**B5 · 挑战隐含假设**（Standard / Deep）
- 需求中是否存在未经验证的假设？
- 是否在定义问题之前就锁定了解决方案？

> **证据优先**：所有分析必须基于实际信息。信息不足时明确标注，不得凭空假设。

### 阶段 5 · 需求分类与路径推荐

基于 JTBD 分析结果，判断需求类型：

| 需求分类 | 判断依据 | 推荐路径 |
|---------|---------|---------|
| 新增业务能力 | 需要新增对象动作、状态流转、业务规则 | ①→②→③→④→⑤ |
| 扩展现有能力 | 在现有动作上新增场景/规则/字段 | ①→②→③→⑤ |
| Bug 修复 · 规则类 | 校验/规则/金额/税率/状态/字段/合规/计算行为与预期不符 | ①→②(lite)→③(条件)→⑤ |
| Bug 修复 · 体验类 | 页面/显示/样式/交互/文案/按钮等 UI 问题 | ①→⑤ |
| Bug 修复 · 技术类 | 超时/崩溃/性能/接口/500 等系统错误 | ①→⑤ |
| 体验优化 | 界面交互改善，不涉及业务逻辑 | ①→④→⑤ |
| 配置变更 | 新增枚举值/参数调整 | ①→②→⑤ |
| 性能优化 | 响应时间/吞吐量改善 | ①→⑤ |

**Bug 子分类判定规则：**

```
Bug 描述中命中以下关键词 → 规则类 Bug：
  · 校验 / 规则 / 金额 / 税率 / 合计 / 计算 / 状态 / 字段约束 / 合规 / 精度 / 误差
  · 应该拦截但没拦截 / 应该通过但被拦截
  · 与本体 RulePackage / ValueSets / ActionTypes.Guard 相关

否则 → 体验类 Bug 或技术类 Bug（直接走 ①→⑤）
```

**产物头部新增字段（Bug 类需求必填）：**

```yaml
bug-subtype: rule | ux | tech
# rule：规则类 Bug，需走 ②(lite) 本体匹配
# ux：体验类 Bug，直接走 ⑤
# tech：技术类 Bug，直接走 ⑤
```

### 阶段 5.5 · 业务需求列表（仅 Standard / Deep）

> **仅在需求分类为「新增能力 / 扩展能力 / 配置变更」时输出。**

从 Functional Jobs 中提炼业务需求列表：

- 每条业务需求必须为「动词+宾语+情境」格式
- 这是 ② 本体查询与映射的直接输入
- 按优先级排序
- **不涉及本体概念**（不出现 ACT-XX、VL4、RulePackage 等术语）
- **不做系统级功能拆解**（不拆到单条校验规则级别）
- 粒度标准：每条业务需求应对应一个**可独立理解的业务场景**

**示例（正确粒度）**：

| #   | 业务需求                   | 优先级 |
| --- | ---------------------- | --- |
| 1   | 开具成品油蓝字发票（正数发票全生命周期）   | P0  |
| 2   | 开具成品油红字发票（负数流程，含红字确认单） | P0  |
| 3   | 校验成品油发票合规性（成品油专项规则）    | P0  |

**反例（粒度过细，不应出现在 ①）**：
- ❌ "校验成品油发票商品编码归类为01" — 这是单条校验规则，属于 ② 的拆解范围
- ❌ "在 ACT-04 中按需加载 VL4 规则" — 这涉及本体概念，属于 ② 的映射范围

> **Fast Lane 注意**：Fast Lane 的多条业务需求描述在 ② 中会合并为 1 个子需求，无需在此强制拆分为独立条目。描述清楚实现的不同角度（校验逻辑、错误处理、适用范围等）即可。

### 阶段 6 · 优先级判断与用户确认

**6.1 给出优先级判断**
- 哪个 Job 最关键？为什么？
- 哪个 Pain 最严重？
- 如果只能做一件事，建议做什么？

**6.2 展示分析结果**

向用户展示完整 JTBD 分析报告（按 [templates/template.md](templates/template.md) 格式）。

**6.3 用户确认**

明确询问：
- JTBD 分析是否准确？
- 需求分类和推荐路径是否合理？
- 业务需求列表是否完整？（注意：不要遗漏业务闭环，如开票必然包含正数+负数）
- 是否需要补充用户访谈？

**示例（错误，禁止）：**
- `requirement-analysis.md` ❌
- `analysis.md` ❌
- `需求分析.md` ❌
- 任何其他名称 ❌

**⚠️ 产物数量强制约束（不可违反）：**

本 skill 只允许写入 **1 个文件**，禁止创建任何其他文件：
- Fast Lane：`{id}_需求分析报告.md`（精简版）× 1

- **Fast Lane**：精简版（≤400 字），仅含分流结论 + 需求分类 + 处置建议
- **Standard / Deep Lane**：完整版（按 [templates/template.md](templates/template.md) 格式）

## 输出契约（下游如何消费）

### 产物头部必填字段

```yaml
lane: fast | standard | deep
priority: P0 | P1 | P2 | P3
priority-rationale: {判定依据}
override-reason: {若有通道/级别覆盖}
bug-subtype: rule | ux | tech   # 仅 Bug 类需求填写
interaction-type-hint: Page | API | Batch | Event | Other  # 粗判，② 精判后以特性清单为准
skill-exit-status: DONE | BLOCKED
skill-exit-detail: {补充说明，BLOCKED 时必填}
```

### skill-exit-status 定义

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `DONE` | 正常完成 | 产物已写入，用户已确认 |
| `BLOCKED` | 无法继续 | 输入缺失、用户拒绝确认、信息严重不足无法分析 |

### 下游消费映射

| 下游 skill | Fast 消费 | Standard / Deep 消费 |
|---|---|---|
| ② ontology-context | 仅 bug-subtype=rule 时消费（lite 模式，输出本体映射摘要） | 业务需求列表（动词+宾语+情境） |
| ⑤ generate-prd | 需求分析报告（精简版） | 需求分析报告全文 |

## 重要约束

1. **与本体解耦**：本 skill 不依赖 Invoice Ontology，不拉取本体文件，不使用本体术语（ACT-XX、VL4、RulePackage 等）
2. **证据优先**：所有分析必须基于实际信息，不得凭空假设
3. **信息不足时必须标注**：主动说明哪些分析需补充用户研究
4. **用户确认为必须步骤**：阶段 4 不可跳过
5. **不扩展范围**：需求拆解以实际描述为准，不主动补充"顺便加上"
6. **不替用户做 MVP 裁剪**：如果需求描述说"支持成品油发票开具"，不应自行假设"红票本期不做"——开票与红冲是发票最小闭环，应完整列出后由用户确认范围
7. **业务需求列表格式**：必须为「动词+宾语+情境」，这是 ② 的输入契约
8. **业务需求粒度**：每条对应一个可独立理解的业务场景，不拆到系统级功能或单条校验规则
9. **分流先于分析**：阶段 2 必须在阶段 4 之前完成
10. **分流必须显性**：每份产物头部必须写明 lane 和判定依据
11. **Fast 不建子 issue**：Fast 通道下禁止创建 Linear 子 issue
12. **Fast 不得跑完整 JTBD**：Fast 仅执行 B1，不得"顺便分析一下"
