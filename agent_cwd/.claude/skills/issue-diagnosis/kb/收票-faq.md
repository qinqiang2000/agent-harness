# 收票类 FAQ

> 涵盖：发票查验失败、影像识别异常、收票状态异常、PDF/OFD 下载问题等。

## 合规校验通用诊断步骤

> 适用于 Q7～Q14 所有合规校验类型

1. 通过日志定位 traceId（参考 [references/log-analysis.md](../references/log-analysis.md) 的「合规校验：通过发票号码定位 traceId」）
2. 用 traceId 查完整链路日志，在链路日志中按各 FAQ 的「链路日志分析关键字」提取字段（这些关键字仅用于分析已查到的日志内容，**不用于构建 searchWordList 搜索**）
3. 根据提取字段输出结论

## Q1: 版式文件没有下载次数

1. 如果是乐企开票，乐企的规范并不要求有下载次数。
2. 如果是非乐企的：
   - 我们从税局下载失败，会先自己生成一个用于展示票面数据，这种也是没有下载次数的。
   - 第一次从税局下载的没有次数，税局服务商资源不够，有时生成会不完整。

---

## Q2: 点击PDF/OFD文件下载链接显示正在下载中。"xxx文件税局未生成，请一段时间后再试"

可能是当时税局波动，未成功下载。可以将最后的 type 改为 3，先下载 xml。下载 xml 成功后会自动触发下载 PDF 和 OFD。

---

## Q3: 查验报错：地方税局网络不稳定，请稍后再试

可以先到国税查验平台查验试试。如果查验平台也报错说明电子税局暂时有问题，需要稍候再试。如果查验平台可以查到，则到收票值班群反馈。

## Q4: 同一发票查验状态不一致

> 相似问题: 发票状态忽正常忽红冲、查验结果不一致、不同时间查验结果不同、缓存状态与税局不同步

**现象**：
- 同一发票号在短时间内多次查验，返回不同状态（如先返回"正常"，后返回"全额红冲"）
- 国税平台显示已红冲，但系统返回正常状态
- 不同租户查询同一发票返回结果不同

**根因**：
缓存 key 包含租户表下标（tableIndex），格式为 `checkData:{invoiceNo}{tableIndex}`。不同租户或不同查询请求使用不同的 tableIndex，导致：
1. 缓存 key 不匹配，无法命中已有缓存
2. 从数据库查出的是旧状态（未同步税局最新状态）
3. 实时查验时才能获取税局最新状态

**日志特征**：
```
查验发票redisKey[checkData:261420000001211879265]  // 带下标的 key
查验发票redisKey main[checkData:26142000000121187926]  - [true]  // 不带下标的 key
以不带下标的key再查一次[checkData:26142000000121187926] - [null]  // 缓存未命中
```

**解决方案**：
1. 确认发票在国税平台的最新状态（权威来源）
2. 清理系统中该发票的所有缓存（包括带下标和不带下标的 key）
3. 重新查验发票，强制实时调用税局接口获取最新状态
4. 如需持久化修复，需要更新数据库中的发票状态记录

**数据库验证**：
查询 invoice库 `t_vat_ele_invoice` 表，检查该发票的 `invoice_status` 字段是否与税局平台一致。

## Q5: 重复报销 - 发票被哪个企业/单据报销了

> 相似问题: 发票已被报销、该发票已关联报销单、重复报销提示、导入发票提示重复

**现象**：
- 发票导入时提示"重复报销"或"该发票已被报销"
- 用户想知道是哪个企业或哪张报销单占用了这张发票

**根因**：
系统在收票合规校验阶段，查询 `t_bill_expense_relation` 表，发现该发票流水号（`fserial_no`）已关联到状态为审批中（30）、已通过（60）或已入账（65）的报销单。

**链路日志分析关键字**：`重复报销msg`
**提取字段**：`serialNo`/`fserial_no`（发票流水号）

> 诊断流程见「合规校验通用诊断步骤」

找到 traceId 后，从合规性校验日志中提取发票流水号，再查询数据库。

**数据库验证**（数据源：prod-invoice）：
查询 `t_bill_expense_relation` 表，使用「合规校验-重复报销查询」SQL 模板，按 `fserial_no` 查找状态为 30/60/65 的记录，返回报销单号、企业ID、状态、创建时间。

查到 `fclient_id` 后，**追加查询 cms 库 `t_ou_company` 表**，获取企业名称（`fname`）和税号（`ftax_no`），在结论中用企业名称替代 clientId 展示。SQL 模板：「重复报销-企业名称查询」。

**fstatus 含义**：
- 30 = 审批中
- 60 = 已通过
- 65 = 已入账

**输出结论示例**：
> 该发票已被企业 [ftax_no] [name] 的报销单 [fexpense_num] 占用，当前状态为「已通过」。

## Q6: 个人发票校验 - 为什么这张票被当作个人发票

> 相似问题: 个人发票拦截、个人抬头、个人票、buyerName 校验失败、购方名称不合法、购方税号为空

**现象**：
- 发票被合规校验拦截，提示"个人发票"或"购方为个人"
- 用户不理解为什么这张发票被判定为个人发票

**根因**：
系统通过 `VerifyUtil.isPersonInvoice` 方法判断发票是否为个人发票，满足以下任一条件即判定为个人发票：

| 条件 | 说明 |
|------|------|
| 购方名称为空 | `buyerName` 字段为空或 null |
| 购方名称含电话号码 | 名称中包含手机号或座机号格式 |
| 购方名称以"（个人）"或"(个人)"结尾 | 字面匹配 |
| 购方税号为空 且 名称长度 ≤ 5 | 两个条件同时满足 |
| 购方税号不符合18位统一社会信用代码规则 | 税号格式不合法 |
| 名称含关键词 | 如"个人"、"自然人"等 |

**票种特殊处理**：
- 通用机打发票（票种=7）、电子发票（票种=23）：参与个人发票校验
- 财政电子票据（票种=25）：**不参与**个人发票判断，直接跳过
- 非可查验票种：跳过校验

**链路日志分析关键字**：`buyerName[`、`通用机打发票、电子发票参与个票校验`、`财政电子票据不参与个票判断`、`非可查验票种不参与个票判断`
**提取字段**：`buyerName`（购方名称）、`buyerTaxNo`（购方税号）、`invoiceType`（票种）

> 诊断流程见「合规校验通用诊断步骤」

从合规性校验日志中提取 buyerName/buyerTaxNo，按上表逐条核查触发条件，并确认票种是否参与校验。

**输出结论示例**：
> 该发票（票种=23）购方税号为空且购方名称"张三"长度≤5，触发个人发票判断条件4，被系统拦截。

## Q7: 购方抬头不一致

> 相似问题: 抬头不一致、发票抬头错误、购方名称不匹配、抬头校验失败、买方名称不对

**提取字段**：`buyerName`（票面购方名称）、`ghfmc`（企业配置名称）、`buyerTaxNo`

> 诊断流程见「合规校验通用诊断步骤」

**结论示例**：发票购方名称"北京某科技有限公司"与企业配置名称"北京某科技有限公司（北京）"不一致，触发拦截。

## Q9: 查验不通过

> 相似问题: 查验失败、查验不通过、发票状态异常、查验返回错误、发票无法查验

**根因**：`CheckStatusImpl` 检查发票查验状态，非成功状态（非"0"/"0000"）则拦截（errorLevel=1）。常见原因：发票信息有误、税局接口异常、发票不存在。

**链路日志分析关键字**：`errKey:`、`checkDescription:`（来自 `CheckStatusImpl` 的 `log.info`）
**提取字段**：`errKey`（错误码）、`checkDescription`（错误描述）

> 诊断流程见「合规校验通用诊断步骤」

**结论示例**：发票查验返回错误码"FPCX9999"，描述为"发票信息不存在"，查验不通过，系统拦截该发票。

## Q10: 发票已红冲/作废

> 相似问题: 红冲发票、作废发票、部分红冲、红冲待确认、发票状态红冲、发票被作废

**根因**：`RedInvoiceImpl` / `InvalidInvoiceImpl` 检查 `invoiceStatus` 字段：
- 全额红冲 → errorLevel=8；部分红冲 → errorLevel=26；红冲待确认 → errorLevel=27；作废 → errorLevel=7

**链路日志分析关键字**：合规性校验日志中无专属日志，通过 `errorLevel` 值（7/8/26/27）和 `verifyResult` 字段判断
**提取字段**：`invoiceStatus`（发票状态，从 `verifyResult` 或查验结果中提取）

> 诊断流程见「合规校验通用诊断步骤」

**结论示例**：该发票当前状态为"全额红冲"（invoiceStatus=红冲），系统拦截，无法报销。

## Q11: 超过报销期限

> 相似问题: 超期发票、跨年发票、报销期限超出、两年期限、发票过期、开票日期太早

**根因**：`ExpenseTwoYearImpl` 比对发票开票日期与当前日期：
- 超过跨年期限（`twoYearMon` 配置，通常24个月）→ errorLevel=15
- 超过报销期限（`expenseDeadline` 配置，企业自定义天数）→ errorLevel=14

**链路日志分析关键字**：合规性校验日志中无专属日志，通过 `errorLevel` 值（14/15）和 `verifyResult` 字段判断；可搜索 `根据发票流水号数组查询发票信息` 获取 `invoiceDate`
**提取字段**：`invoiceDate`（开票日期，从发票信息中提取）、`twoYearMon`/`expenseDeadline`（从合规配置日志 `覆写后的合规性校验 config` 中提取）

> 诊断流程见「合规校验通用诊断步骤」

**结论示例**：发票开票日期为 2022-01-15，距今已超过 24 个月（企业配置跨年期限是{期限或天数}），触发超期拦截。

## Q12: 电子发票未上传源文件

> 相似问题: 未上传源文件、缺少原件、电子发票文件缺失、OFD/PDF未上传、需要上传原始文件

**根因**：`WithoutFileImpl` 检查电子发票是否有原始文件（`fileType=1`），无则拦截（errorLevel=3）。仅对电子发票生效，通用机打发票（票种=23）和区块链发票不参与此校验。

**链路日志分析关键字**：`标记新逻辑，非原件 state`、`标记旧逻辑，非原件`、`已经是原文件，不做校验`
**提取字段**：`state`（originalState）、`serialNo`、`fileType`

> 诊断流程见「合规校验通用诊断步骤」

**结论示例**：该电子发票（票种=10）未上传 OFD/PDF 源文件（fileType 非 1），需重新上传原始电子发票文件。

## Q13: 销方黑名单/敏感词/违法纳税人

> 相似问题: 黑名单拦截、敏感词拦截、违法纳税人、销方黑名单、发票明细敏感词、销方名称违规

**根因**：三类校验，任一触发即拦截：
- **销方名称黑名单**（`FilterSalerNameImpl`）：`salerName` 在企业配置黑名单中 → errorLevel=13
- **发票明细敏感词**（`VerifyKeyImpl`）：`goodsName` 含配置敏感词 → errorLevel=12
- **税局违法纳税人**（`VerifyCompanyBlacklistImpl`）：`salerTaxNo` 在税局公示违法纳税人名单中 → errorLevel=19

**链路日志分析关键字**：`销方名称黑名单发票校验-`、`校验敏感词--不通过`、`校验销方黑名单`
**提取字段**：`salerName`（销方名称）、`salerTaxNo`（销方税号）、`filterKey`（匹配的关键词/敏感词）

> 诊断流程见「合规校验通用诊断步骤」

**结论示例**：发票商品名称"娱乐消费"命中企业配置敏感词"娱乐"，触发发票明细敏感词校验拦截（VerifyKeyImpl）。

## Q14: 发票连号

> 相似问题: 连号发票、发票号码连续、发票号码相邻、连号拦截、发票号码范围

**提取字段**：`invoiceNo`（发票号码）、`invoiceCode`（发票代码）、`invoiceType`（票种）、连号范围列表

> 诊断流程见「合规校验通用诊断步骤」

**结论示例**：发票号码 01093447 与同批次发票号码 01093448 相差 1，在连号检测范围（range=5）内，触发发票连号校验拦截。

## Q15: 绑定邮箱提示"邮箱已被绑定"

> 相似问题: 邮箱已绑定、邮箱重复绑定、邮箱绑定失败

**根因**：该邮箱在 `t_mail_user_relation` 表中已存在绑定记录，但记录中的 `fopen_id` 与当前用户的 openId 不匹配，即同一邮箱被其他账号绑定过。

**日志关键字**：`["发送邮件验证码sendVerifyCode param", "<邮箱地址>"]`
**提取字段**：邮箱地址、当前用户 openId

**数据库验证**（数据源：prod-invoice）：
```sql
SELECT fid, fmail_user, fopen_id, fclient_id, fcreate_time, fupdate_time, fclient_type
FROM t_mail_user_relation
WHERE fmail_user = '{邮箱}'
```
对比 `fopen_id` 与当前用户 openId，判断是否为同一用户绑定。

**结论示例**：
> 邮箱 xxx@xxx.com 已被 openId 为 `abc123` 的账号绑定（fclient_id=xxx），与当前用户 openId `def456` 不一致，提示"邮箱已被绑定"。
